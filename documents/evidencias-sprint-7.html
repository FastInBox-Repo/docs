3.1 Security headers em toda resposta

BACK | HARDENING
Middleware: main.ts Escopo: todas as respostas Status: 200 OK

Todo response carrega os cabecalhos de seguranca aplicados no bootstrap. O dump abaixo foi capturado com curl -D - contra /health. O Strict-Transport-Security e emitido apenas em producao e por isso esta corretamente ausente em desenvolvimento.

Como foi testado

  1. Dump dos headers via curl -D - http://localhost:4001/health.
  2. Conferencia dos 5 cabecalhos obrigatorios em dev.
  3. Validacao de que HSTS aparece somente com NODE_ENV=production.

Como voce testa

  1. Suba o backend com npm run start:dev.
  2. Rode curl -D - http://localhost:4001/health -o /dev/null.
  3. Confira os cabecalhos no topo da resposta.
$ curl -sS -D - http://localhost:4001/health -o /dev/null
HTTP/1.1 200 OK
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
Referrer-Policy: no-referrer
X-DNS-Prefetch-Control: off
Cross-Origin-Opener-Policy: same-origin
Content-Type: application/json; charset=utf-8
Content-Length: 71
# Strict-Transport-Security ausente em dev (emitido apenas em producao)

3.2 Rate limiting por IP (janela fixa)

BACK | RATE LIMIT
Guard: rate-limit.guard.ts (APP_GUARD) Limite testado: RATE_LIMIT_MAX=3 Excedente: 429 RATE_LIMITED

Burst controlado contra o servidor iniciado com RATE_LIMIT_MAX=3: as 3 primeiras requisicoes passam (200) e as seguintes sao rejeitadas (429) com body estruturado. As rotas /health e /ready sao isentas via @SkipRateLimit() e respondem 200 mesmo apos o burst.

Como foi testado

  1. Spec Jest rate-limit.guard.spec.ts cobrindo janela, contagem e isencao.
  2. Burst manual de 6 requisicoes com RATE_LIMIT_MAX=3.
  3. Confirmacao de que /health pos-burst retorna 200 (exempt).

Como voce testa

  1. Suba com RATE_LIMIT_MAX=3 npm run start:dev.
  2. Dispare 6 chamadas em sequencia ao mesmo endpoint.
  3. Observe 200 nas 3 primeiras e 429 nas 3 ultimas.
$ for i in $(seq 1 6); do \
    code=$(curl -s -o /dev/null -w "%{http_code}" http://localhost:4001/admin/diagnostics); \
    echo "req$i -> $code"; \
  done
req1 -> 200
req2 -> 200
req3 -> 200
req4 -> 429
req5 -> 429
req6 -> 429

# body real da 4a requisicao (HTTP 429)
$ curl -s http://localhost:4001/admin/diagnostics | jq
{
  "statusCode": 429,
  "code": "RATE_LIMITED",
  "message": "Too many requests. Please slow down.",
  "traceId": "8f2c1a9e-7b40-4d11-9c3a-2e6d5b0f4a87"
}

# /health permanece 200 apos o burst (isento via @SkipRateLimit)
$ curl -s -o /dev/null -w "/health -> %{http_code}\n" http://localhost:4001/health
/health -> 200

3.3 Readiness para orquestradores

BACK | READINESS
Endpoint: GET /ready Auth: publico Status: 200 OK

Endpoint de readiness usado por orquestradores e monitores externos para decidir se a instancia pode receber trafego. Retorna o status agregado, o resultado do check de seed e o uptime em milissegundos. Publico e isento de rate limit.

$ curl -s http://localhost:4001/ready | jq
{
  "status": "ready",
  "checks": {
    "seed": "ok",
    "uptimeMs": 423
  }
}
