3. Estrategia de branching

ResponsaveisBackThiago Lucas AlvesArquiteturaGabriel Pahl

Modelo adotado: Trunk-Based Light com main protegida e feature branches curtas. Hotfix isolado quando producao quebra.

BranchOrigemDestinoPolitica
main--Protegida. PR obrigatorio. Build verde + 1 review aprovada.
feature/<area>-<descricao>mainmain via PRCurta (< 3 dias). Squash merge.
fix/<area>-<descricao>mainmain via PRCorrecao pontual sem mudanca de contrato.
hotfix/<versao>tag de releasemain + tag patchApenas para producao quebrada. Documentar em sprint review.
chore/docs-<tema>mainmain via PRAtualizacao de documentacao sem efeito em runtime.

ResponsaveisArquiteturaGabriel Pahl

Versionamento SemVer 2.0.0 com tag por sprint encerrada.

8. Gerenciamento de segredos

ResponsaveisBackThiago Lucas AlvesSegurancaGabriel Pahl
  • Vercel Environment Variables para front (NEXT_PUBLIC_API_URL).
  • Heroku Config Vars para back (DATABASE_URL, JWT_SECRET, REFRESH_SECRET, HMAC_WEBHOOK_KEY).
  • GitHub Secrets para Actions (deploy tokens, Playwright API key).
  • .env.example versionado em cada repo; .env real ignorado por .gitignore.
  • Rotacao prevista a cada release MINOR e obrigatoria em hotfix de credencial.
Nenhum segredo deve trafegar em commit. Pre-commit hook (gitleaks) escaneia push antes de subir.

10. Controle de mudanca

ResponsaveisArquiteturaGabriel PahlBackThiago Lucas Alves
  1. Issue criada no GitHub Project com area, prioridade, owner e sprint.
  2. Branch derivada da main seguindo nomenclatura padrao.
  3. PR aberto contra main com descricao, checklist e link da issue.
  4. CI roda lint, typecheck, build e smoke E2E.
  5. Review obrigatoria de outro membro do time. Bloqueio se houver red flag.
  6. Squash merge na main com commit semantico.
  7. Deploy automatico para preview/production conforme branch.
  8. Status atualizado no GitHub Project para Done.

11. Backup e recuperacao

ResponsaveisBackThiago Lucas Alves
  • Codigo: redundancia nativa do GitHub. Clones locais por desenvolvedor.
  • Banco: backup automatico diario via Neon (retencao 7 dias). Point-in-time recovery 24h.
  • Documentacao: versionada no Git + replicada em GitHub Pages.
  • Releases: tags imutaveis garantem snapshot reproduzivel.

12. Auditoria e rastreabilidade

ResponsaveisQAGabriel Pahl

Cada artefato tem trilha auditavel:

  • Requisito (ERS) -> historia de usuario -> issue GitHub -> PR -> commit -> deploy -> evidencia.
  • Logs operacionais persistidos no modulo audit do back (production_events).
  • Sprint reviews documentadas em sprint-review.html.
  • Backlog vivo no GitHub Project com 78 itens.
