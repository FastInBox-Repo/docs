3. Mad / Sad / Glad

ResponsaveisArquiteturaGabriel PahlBackThiago Lucas AlvesFrontJoao Vitor Marra

Glad - alegrias

  • Hub de documentacao saiu na S1 e nunca dessincronizou do codigo.
  • Migracao de persistencia local para PostgreSQL na S3 sem regressao.
  • Autenticacao server-side com refresh rotation funcionou de primeira.
  • Identidade visual neo-brutal foi adotada pelo time sem retrabalho.
  • 22 specs Playwright pegando regressoes antes do PR merge.

Sad - tristezas

  • Sprint 2 ficou curta (8 itens) por falta de planning detalhado.
  • Estimativas iniciais erraram em ate 30% nos itens novos de NestJS.
  • Pair programming foi raro - codigo critico ficou em head solo.
  • Daily Scrum nem sempre virou registro escrito em ALM.

Mad - frustracoes

  • Configuracao de seed do banco quebrou em 2 PRs por dependencia oculta.
  • Conflitos de merge em arquivos de docs gerados sem politica clara.
  • Webhook do gateway demorou 2 dias por documentacao confusa do provedor.

4. Start / Stop / Continue

ResponsaveisArquiteturaGabriel Pahl

Start - comecar

  • Daily Scrum registrada em audio + transcrita em ALM no mesmo dia.
  • Pair programming em rotas criticas (auth, payments, audit).
  • Refinement formal 48h antes do planning para reduzir estimativa errada.
  • Politica de PR menor (<= 400 linhas) para review mais rapida.

Stop - parar

  • Estimar itens novos por analogia sem spike de validacao.
  • Misturar refactor com feature no mesmo PR.
  • Esperar fim de sprint para atualizar docs.

Continue - manter

  • Conventional Commits em todos os repos.
  • Trunk-based light com main protegida e PR obrigatorio.
  • Evidencias visuais (prints + video) por sprint review.
  • GitHub Project como fonte unica de verdade.
  • Documentacao tecnica em GitHub Pages atualizada por sprint.
