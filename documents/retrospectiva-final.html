1. Contexto: ciclo completo S1 -> S7

ResponsaveisArquiteturaGabriel Pahl

O FastInBox nasceu como MVP transacional para nutricionistas montarem planos de marmitas, pacientes confirmarem pedidos por link publico e cozinhas operarem um kanban de producao. Ao longo de 7 sprints o produto saiu de um front navegavel sem backend ate uma plataforma com gateway de pagamento, governanca administrativa, escala comercial e operacao endurecida para producao.

Este documento encerra o ciclo de entrega no marco da Aula 20 (Revisao da Sprint 7 + Retrospectiva + 3a Avaliacao Parcial + Apresentacao Final). Ele complementa a Retrospectiva dos Sprints 1-3, cujas 7 acoes de melhoria sao reavaliadas na secao 5, e consolida o aprendizado do time inteiro do ciclo.

Estado ao fechamento (Aula 20):

  • 7 sprints entregues, todas as duas ultimas (S6 e S7) com 7/7 itens em Done.
  • Backend NestJS modular com GET /ready, rate limiting, security headers e smoke de release verde.
  • Jornadas cobertas ponta a ponta: nutricionista, paciente (incl. autoatendimento whitelabel), cozinha, admin.
  • 0 defeitos bloqueantes em aberto. Plataforma em estado de go-live.

3. Mad / Sad / Glad consolidado do ciclo

ResponsaveisArquiteturaGabriel PahlBackThiago Lucas AlvesFrontJoao Vitor Marra

Glad - alegrias

  • Ciclo fechou em go-live com S4, S5, S6 e S7 todas em 7/7 Done.
  • Hub de documentacao nunca dessincronizou do codigo em 7 sprints.
  • Gateway transacional com webhook HMAC subiu sem regressao na S4.
  • Assinaturas recorrentes geram pedidos automaticos via runDue() no boot.
  • Previsao de demanda com banda de confianca bateu o invariante expectedMeals = fromHistory + fromSubscriptions.
  • Autoatendimento whitelabel renderiza marca da clinica com contraste calculado por luminancia.
  • Smoke de release npm run smoke fechou em 8 passed, 0 failed antes do go-live.

Sad - tristezas

  • Sprint 2 ficou curta (8 itens) por falta de planning detalhado.
  • Estimativas iniciais de NestJS erraram em ate 30% nos itens novos.
  • Cobertura de testes do backend estabilizou em 6 suites / 30 testes; faltou subir mais cedo.
  • Stores em memoria no backend agilizaram o MVP mas adiam a persistencia definitiva pos-go-live.
  • UX cross-device foi revisado tarde, ja na S7, e nao a cada sprint.

Mad - frustracoes

  • Seed do banco quebrou em 2 PRs no inicio do ciclo por dependencia oculta.
  • Webhook do gateway demorou 2 dias por documentacao confusa do provedor.
  • Conflitos de merge em docs gerados sem politica clara no comeco do ciclo.
  • Ajustar rate limiting sem ruido nos endpoints exemptos (/health, /ready) custou idas e vindas.

4. Start / Stop / Continue

ResponsaveisArquiteturaGabriel Pahl

Start - comecar

  • Migrar os stores em memoria para PostgreSQL persistente no pos-go-live.
  • Revisao UX cross-device como ritual fixo de cada sprint, nao so no fechamento.
  • Smoke de release agendado em CI a cada deploy em main, nao apenas manual.
  • Monitoramento externo consumindo /ready e /admin/diagnostics com alerta.

Stop - parar

  • Adiar cobertura de testes para o final da sprint.
  • Estimar itens novos por analogia sem spike de validacao.
  • Misturar refactor com feature no mesmo PR.
  • Tratar observabilidade como item de hardening pontual em vez de continuo.

Continue - manter

  • Pair programming em rotas criticas (auth, payments, audit).
  • Conventional Commits e trunk-based light com main protegida.
  • Docs atualizadas no mesmo PR da feature como gate de release.
  • Evidencias visuais (prints + curl real) por sprint review.
  • GitHub Project como fonte unica de verdade do escopo.
