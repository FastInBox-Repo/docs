FastInBox | Operacao e SRE

Runbook Operacional e Rollback

Entregavel da Sprint 7 (Producao e go-live). Procedimentos operacionais para manter o FastInBox no ar, diagnosticar incidentes por sintoma e executar rollback sem dependencias externas.

Documento dependency-free: todos os passos usam apenas git, o CI de cada repo e os scripts npm ja versionados (npm run smoke). Health e readiness expostos pela API permitem checagem por qualquer monitor externo.

Premissa operacional importante: o backend usa stores em memoria e roda seed a cada boot. Um restart ou novo deploy reinicializa o dataset de demonstracao. Nao ha banco persistente a migrar em rollback, o que torna o procedimento puramente baseado em git + CI.

2. Health e readiness

ResponsaveisBackThiago Lucas AlvesSREGabriel Pahl

A API expoe dois probes publicos e isentos de rate limit (decorador @SkipRateLimit()), pensados para load balancers e orquestradores.

EndpointUsoAuthRate limit
GET /healthLiveness probe. O processo esta vivo e respondendo.PublicoIsento
GET /readyReadiness probe. O servico bootou e o seed concluiu antes de receber trafego.PublicoIsento

O /ready reporta o status do seed e o uptime em milissegundos. Enquanto checks.seed for pending, o readiness deve ser tratado como nao pronto e o trafego retido.

$ curl -s http://localhost:4001/ready | jq
{
  "status": "ready",
  "checks": {
    "seed": "ok",
    "uptimeMs": 423
  },
  "timestamp": "2026-06-23T11:02:18.704Z"
}

Regra de uso: o load balancer roteia para o backend apenas quando /ready retorna 200 com status:"ready" e checks.seed:"ok". O /health deve responder 200 mesmo durante um burst de rate limit, justamente por ser isento, confirmando que o processo continua vivo.

3. Observabilidade

ResponsaveisBackThiago Lucas AlvesArquiteturaGabriel Pahl

O painel de saude operacional e o GET /admin/diagnostics (role admin). Em uma unica chamada ele agrega o estado dos modulos de negocio: orders, payments, audit e, desde a Sprint 6, subscriptions e selfService.

$ curl -sH "Authorization: Bearer $TOKEN" http://localhost:4001/admin/diagnostics | jq
{
  "status": "ok",
  "modules": {
    "orders":        { "status": "ok", "total": 7, "delivered": 3 },
    "payments":      { "status": "ok", "intents": ..., "approved": ..., "failed": ... },
    "audit":         { "status": "ok", "totalEntries": ... },
    "subscriptions": { "status": "ok", "total": 3, "active": 3, "paused": 0, "cancelled": 0 },
    "selfService":   { "status": "ok", "codes": 1 }
  },
  "timestamp": "2026-06-23T11:05:41.000Z"
}

Leitura recomendada por bloco durante um incidente:

  • orders - volume total e entregues. Queda abrupta de total indica falha de criacao de pedidos.
  • payments - intents aprovadas vs falhas e webhooks processados/rejeitados. failed ou webhooksRejected subindo sinaliza problema de pagamento.
  • subscriptions - contagem por status. Util apos rodar a cobranca recorrente para confirmar que nenhuma assinatura entrou em estado inesperado.
  • selfService - quantidade de codigos ativos de autoatendimento.

Para trilha detalhada de eventos use GET /admin/auditoria (role admin), que lista o audit log com outcome por evento, incluindo tentativas negadas (outcome=denied). E a fonte primaria para reconstruir a linha do tempo de um incidente.

4. Resposta a incidentes

ResponsaveisSREGabriel PahlBackThiago Lucas AlvesFrontJoao Vitor Marra

Runbook por sintoma. Identifique a linha que corresponde ao alerta, siga o diagnostico, execute a acao e acione o owner se a acao nao resolver.

SintomaDiagnosticoAcaoOwner
API retornando 5xx Checar GET /health e GET /ready. Se /health nao responde, processo caiu; se /ready mostra seed:"pending", boot incompleto. Conferir /admin/diagnostics e logs do CI/runtime. Reiniciar o processo do backend (reinicia stores + seed). Se reincidir apos o ultimo deploy, executar ROLLBACK do back (secao 5). Thiago Lucas (Back)
Pico de 429 (rate limit) Respostas com code:"RATE_LIMITED". Confirmar que e trafego legitimo vs abuso. /health e /ready continuam 200 por serem isentos. Se legitimo, elevar RATE_LIMIT_MAX e/ou RATE_LIMIT_WINDOW_MS e reiniciar. Se abuso, manter o limite e bloquear o IP na borda. Gabriel Pahl (Seguranca)
Webhook de pagamento falhando No /admin/diagnostics, bloco payments com webhooksRejected ou failed subindo. Cruzar com /admin/auditoria. Causas tipicas: assinatura HMAC invalida ou janela anti-replay (+/- 5 min). Validar segredo HMAC e o relogio do provedor. Confirmar que rawBody esta intacto (sem proxy reescrevendo o corpo). Reenviar o webhook apos correcao. Thiago Lucas (Back)
Build de CI vermelho Pipeline falhando no push em main. Identificar o stage: build, lint ou test. Baseline saudavel: 6 suites / 30 testes, lint exit 0, build limpo. Corrigir e re-push, ou executar ROLLBACK do repo afetado para destravar o deploy enquanto o fix e preparado. Gabriel Pahl (QA)
Pages fora do ar Hub fastinbox-repo.github.io/docs retornando 404 ou versao antiga. Conferir o ultimo workflow de Pages no repo docs. Re-disparar o deploy de Pages (re-run do workflow). Se o conteudo estiver quebrado, ROLLBACK do docs e aguardar o rebuild de Pages. Joao Vitor (Front)

5. Procedimento de ROLLBACK

ResponsaveisBackThiago Lucas AlvesSREGabriel Pahl

O rollback e dependency-free e identico nos tres repos: reverter o merge em main, deixar o CI reconstruir e validar. Como nao ha banco persistente, nao existem migracoes a desfazer.

Criterios de decisao

Inicie rollback quando, apos o ultimo deploy, qualquer condicao abaixo for verdadeira:

  • API em 5xx sustentado e reiniciar o processo nao resolve.
  • npm run smoke falha em qualquer um dos 8 checks (secao 6).
  • Build de CI vermelho bloqueando deploys e o fix nao sai em ate 15 minutos.
  • Regressao funcional confirmada por um owner que afeta um fluxo critico (login, criacao de pedido, pagamento).

Passo a passo (por repo)

  1. Identificar o commit de merge problematico em main: git log --oneline --merges -n 5.
  2. Reverter o merge preservando o historico (sem reescrita): git revert -m 1 <merge_sha>.
  3. Empurrar para main: git push origin main. Isso dispara o CI rebuild (back/front) ou o deploy de Pages (docs).
  4. Aguardar o pipeline ficar verde (build + lint + test no back/front; publicacao de Pages no docs).
  5. Validar a release com npm run smoke contra o servidor (secao 6). Os 8 checks precisam passar.
# Exemplo - rollback do backend
$ git log --oneline --merges -n 5
$ git revert -m 1 <merge_sha>
$ git push origin main          # dispara CI rebuild
# apos pipeline verde:
$ npm run smoke                 # 8 checks contra o servidor
8 passed, 0 failed, 0 skipped

Repita o procedimento isoladamente em cada repo afetado. Front e back rebuildeam pelo CI no push; o docs republica via GitHub Pages.

Janela e responsavel on-call

Rollback pode ser executado a qualquer momento pelo on-call de plantao. A decisao de rollback nao precisa de quorum: o on-call decide e executa, depois comunica os owners. Owner por repo: back Thiago Lucas, front Joao Vitor, docs Gabriel Pahl. O criterio verde do npm run smoke e a confirmacao final antes de declarar o incidente encerrado.

6. Smoke de release

ResponsaveisQAGabriel PahlBackThiago Lucas Alves

O smoke de release roda contra um servidor ja no ar (precisa do backend up) e e o gate verde antes de promover/switchar trafego para a nova versao. Script versionado em back/scripts/smoke.mjs.

# com o backend rodando (ex.: http://localhost:4001):
$ npm run smoke
8 passed, 0 failed, 0 skipped

Sao 8 checks ordenados. A falha de qualquer um e criterio de rollback:

  1. GET /health responde 200.
  2. GET /ready responde 200 com status:"ready" e seed:"ok".
  3. Login admin via POST /auth/login retorna accessToken.
  4. GET /admin/diagnostics responde 200 com os blocos esperados.
  5. Invariantes do forecast (GET /admin/reports/forecast): expectedMeals === fromHistory + fromSubscriptions e lowerBound <= expectedMeals <= upperBound.
  6. GET /subscriptions retorna ao menos 3 assinaturas.
  7. Codigo de autoatendimento invalido em GET /public/self-service/:code retorna 404 SELF_SERVICE_CODE_INVALID.
  8. Codigo de autoatendimento valido retorna o whitelabel da clinica (clinic + nutritionist).

Resultado de referencia da Sprint 7: 8 passed 0 failed, 0 skipped. Criterio verde = os 8 checks passam antes do switch.
