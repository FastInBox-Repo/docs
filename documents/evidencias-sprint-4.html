3.1 Confirmacao do paciente + criacao do intent

BACK | PAYMENT
Status pre: AWAITING_PATIENT_REVIEW Status pos: AWAITING_PAYMENT Provider: MockPaymentProvider

O paciente confirma o pedido e o front (ou um cliente HTTP) requisita o intent transacional. O servico aplica o gate de status, gera providerIntentId e devolve uma URL de checkout. O front redirecionaria o usuario para essa URL no provider real.

Como foi testado

  1. Confirm em pedido com status diferente: rejeitado com 409.
  2. Criacao de intent duplicada para mesmo pedido em pending: retorna o mesmo intent (idempotente).
  3. Confirmacao + intent + simulacao geram 3 eventos consistentes.

Como voce testa

  1. curl -X POST http://localhost:4001/public/orders/<code>/confirm
  2. curl -X POST http://localhost:4001/public/orders/<code>/payment-intent
  3. Anote providerIntentId retornado.

3.2 Aprovacao via simulador (caminho do webhook)

BACK | WEBHOOK
Idempotente por providerEventId Atomico Side effect: PAID + kitchen assigned

O simulador reproduz exatamente o caminho do webhook do provider real. O servico marca o evento como processado, transiciona o pedido para PAID, atribui cozinha via round-robin e grava todos os eventos de auditoria correlacionados.

Como foi testado

  1. Aprovacao com providerIntentId valido: pedido vai para PAID e kitchen e atribuida.
  2. Reenvio do mesmo evento: retorna duplicated:true e nao reprocessa estado.
  3. Aprovacao com providerIntentId desconhecido: retorna status:unknown_intent e marca terminal_failed.

Como voce testa

  1. curl -X POST http://localhost:4001/payments/simulate/<providerIntentId>/approve
  2. Repita o comando para ver idempotencia.
  3. Bata em GET /orders/<id> e veja status PAID.
$ curl -sX POST http://localhost:4001/payments/simulate/mock_96cf6616-.../approve | jq
{ "duplicated": false, "providerEventId": "evt_4af2705e-...", "status": "approved" }

$ curl -sX POST http://localhost:4001/payments/simulate/mock_96cf6616-.../approve | jq
{ "duplicated": true,  "providerEventId": "evt_4af2705e-..." }

$ curl -s -H "Authorization: Bearer $T" http://localhost:4001/orders/$ID | jq '{status, paymentStatus, kitchenId, paidAt}'
{
  "status": "PAID",
  "paymentStatus": "approved",
  "kitchenId": "kitchen-e71ccfdc-A",
  "paidAt": "2026-05-06T16:38:54.232Z"

3.3 Cozinha assume + start-production

BACK | OPS
Role: kitchen Transicao: PAID -> IN_PRODUCTION Trilha auditavel ativa

O usuario da cozinha autentica e ja enxerga o pedido na sua lista. O start-production aplica a transicao validada pela maquina de estados, e o historico do pedido cresce linearmente com cada avanco operacional.

Como foi testado

  1. Listagem por kitchenId retorna o pedido recem-pago.
  2. Tentativa de start-production em pedido em outra clinica: 403.
  3. Tentativa em status diferente: 409.

Como voce testa

  1. Login como cozinha (cozinha@fastinbox.test).
  2. POST /orders/<id>/start-production
  3. Confira GET /orders/<id>/history.
