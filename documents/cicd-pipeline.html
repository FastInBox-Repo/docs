CI - Continuous IntegrationPratica de integrar codigo na branch principal varias vezes ao dia, validado por build automatizado e testes.Cada PR dispara lint + typecheck + build + unit tests via GitHub Actions.
CD - Continuous DeliveryCodigo aprovado na main esta sempre apto a deploy para producao, com aprovacao manual final.Vercel e Heroku geram preview por PR; merge em main libera deploy automatico apos build verde.

Lead Time for ChangesTempo entre commit e producao.FastInBox: mediana 12 min (build + deploy + smoke).
Change Failure RatePercentual de deploys que causam falha em producao.FastInBox: 0% nas Sprints 1-5 (rollback nao necessario).
MTTRMean Time To Recovery em incidente.Sem incidente registrado; estimativa em testes: < 5 min via revert + redeploy.

3. Beneficios concretos para o FastInBox

ResponsaveisArquiteturaGabriel Pahl
  • Risco menor por deploy: diffs pequenos sao mais faceis de revisar e reverter.
  • Feedback rapido: autor descobre regressao em ~10 min, nao em sprint review.
  • Confianca para refatorar: suite de testes serve como rede de seguranca em mudancas grandes (ex: migracao para NestJS na Sprint 3).
  • Docs sincronizadas: diff de /docs faz parte do mesmo PR da feature.
  • Demo sempre disponivel: URL Vercel viva para banca, sem janela de manutencao.
  • Auditoria: cada commit tem hash, autor, build log e link para deploy.
  • Rollback rapido: revert + push restaura producao em < 5 min.

6. Quality gates obrigatorios

ResponsaveisQAGabriel Pahl
  1. Lint zero-error: ESLint configurado com rules estritas. Falha bloqueia PR.
  2. Type check estrito: TypeScript em modo strict. Falha bloqueia PR.
  3. Build verde: next build (front) e nest build (back) precisam terminar sem erro.
  4. Unit tests verdes: Jest 100% verde nos repos com testes.
  5. Smoke E2E verde: 22 specs Playwright executam em preview do front antes do merge.
  6. Review obrigatoria: pelo menos 1 reviewer aprovando o PR.
  7. Sem secrets em commit: .gitignore + gitleaks pre-commit hook.
Qualquer falha em qualquer gate impede merge. main e protegida via GitHub branch protection rules.

11. Conclusao academica

ResponsaveisArquiteturaGabriel Pahl

CI/CD foi adotado desde a Sprint 1 do FastInBox, nao como ferramenta separada, mas como disciplina de engenharia. O resultado mensuravel: 5 sprints fechadas com 0 incidentes de regressao em producao, lead time sub-15 minutos e cobertura de testes que cresceu de 3 specs para 22 specs sem dessincronia.

A combinacao trunk-based + PR obrigatorio + suite E2E gratuita do Playwright em preview deployments e o sweet spot para times pequenos que precisam entregar qualidade alta com sobrecarga operacional baixa. Tooling mais sofisticado (feature flags, canary, blue-green) so faz sentido com volume de usuarios e equipe grande - nenhum dos dois aplicaveis a um MVP academico.
