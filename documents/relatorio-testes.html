2. Objetivo e escopo

ResponsaveisBackThiago Lucas AlvesFrontJoao Vitor MarraArquiteturaGabriel Pahl

Registrar formalmente o ciclo completo de testes executado sobre o sistema FastInBox ao longo das Sprints 1, 2 e 3, garantindo rastreabilidade entre requisitos funcionais (RF001-RF014), casos de teste (TC), evidencias capturadas e defeitos tratados. Esta documentacao serve como fonte unica para revisao academica, auditoria de governanca de produto e baseline para a continuidade da Sprint 4.

Em escopo

  • Autenticacao multi-perfil (nutricionista, paciente, cozinha, admin)
  • CRUD de pacientes vinculado ao nutricionista autenticado
  • Builder de pedido com ingredientes, embalagem e codigo unico
  • Landing publica do paciente com confirmacao pre-pagamento
  • Kanban da cozinha sincronizado em tempo quase real
  • Trilha de auditoria de eventos sensiveis (login, transicoes)
  • Disponibilidade da API (/health) e estabilidade de build

Fora de escopo (sera coberto em PI IV)

  • Pagamento real via gateway (sandbox em execucao na Sprint 4)
  • Webhook de aprovacao com idempotencia HMAC (Sprint 4)
  • Fila operacional agrupada por janela de entrega (Sprint 4)
  • Dashboard administrativo com reporting consolidado (Sprint 5)
  • Testes Beta com clinicas parceiras reais (PI IV)

3. Metodologia e estrategia de teste

ResponsaveisBackThiago Lucas AlvesFrontJoao Vitor MarraArquiteturaGabriel Pahl

Estrategia hibrida orientada a risco com base em ISTQB Foundation Level: combinacao de tecnicas de caixa-preta (particao de equivalencia, transicao de estados, casos de uso) e caixa-branca (cobertura de instrucao em servicos puros). A piramide de testes (Cohn, 2009) fundamenta a alocacao de esforco entre os niveis.

6. Criterios de aceite aplicados

ResponsaveisBackThiago Lucas AlvesFrontJoao Vitor MarraQAGabriel Pahl

Um caso de teste e considerado APROVADO apenas quando todas as condicoes a seguir sao satisfeitas. Caso uma falhe, o caso retorna para correcao sem ambiguidade na triagem.

  1. O fluxo atinge o resultado esperado sem erro bloqueante observavel pelo ator.
  2. As regras de negocio sao respeitadas, em especial as permissoes por perfil (nutricionista, paciente, cozinha, admin).
  3. Os dados sao persistidos de forma consistente entre sessoes/refresh.
  4. A integracao front-back responde com payload aderente ao contrato OpenAPI.
  5. Os builds npm run build em front e back terminam com codigo de saida 0.
  6. Pelo menos uma evidencia objetiva (print, log, payload, video) e anexada ao caso.

15. Glossario

ResponsaveisBackThiago Lucas AlvesFrontJoao Vitor MarraArquiteturaGabriel Pahl
APIInterface de comunicacao entre sistemas (frontend e backend), exposta como endpoints HTTP.
BackendCamada de regras de negocio e dados; no FastInBox e o monolito NestJS.
FrontendCamada visual com a qual o usuario interage; aqui Next.js 15 + React 19.
BuildProcesso de compilacao/verificacao tecnica do projeto (zero erro TS).
Smoke testTeste rapido para validar se o fluxo principal esta funcionando antes de testes extensivos.
E2E (End-to-End)Teste da jornada completa do usuario, do inicio ao fim, sem mocks intermediarios.
KanbanQuadro visual de status das atividades / pedidos por estagio operacional.
Server-sideExecucao e persistencia realizadas no servidor; oposto a fixtures locais.
EndpointURL de acesso a uma funcionalidade da API (ex: POST /auth/login).
Status code 403HTTP "Forbidden": acesso negado por regra de permissao.
Criterio de aceiteCondicao que define quando um item pode ser considerado aprovado.
EvidenciaProva objetiva de execucao / resultado (print, log, video, payload, relatorio).
RF / RNFRequisito Funcional / Nao Funcional, definidos na ERS do projeto.
RTMRequirements Traceability Matrix; vincula RF a casos de teste.
DREDefect Removal Efficiency; razao entre defeitos encontrados e corrigidos no proprio ciclo.
HMACHash-based Message Authentication Code; garantia de autenticidade de webhooks.
IdempotenciaPropriedade de uma operacao produzir o mesmo efeito mesmo repetida varias vezes.
Pass ratePercentual de casos de teste aprovados sobre o total executado.
