Sprint 3 Planning

Periodo planejado: Sprint 3 | Status: Entregue com 20 itens em Done e revisao final aprovada

Sprint de transicao do MVP demonstravel para plataforma com persistencia real. Foca na arquitetura back-end NestJS, banco PostgreSQL, autenticacao/autorizacao server-side e sincronizacao do kanban da cozinha com dados reais.

Autor: Thiago Lucas | Ultima atualizacao: 14/05/2026

1. Contexto da sprint

A Sprint 3 parte de um MVP estavel em front-end e auditado pela Sprint 2. O desafio central e eliminar a dependencia de estado local: todo o dominio critico (usuarios, pacientes, pedidos, status operacional) precisa migrar para API real, com autenticacao server-side, autorizacao multi-perfil e historico de status auditavel. E a sprint mais tecnica do ciclo: destrava tudo o que vem depois (pagamento, admin, escala).

2. Objetivo central

Consolidar o back-end NestJS com entidades core persistidas em PostgreSQL, expor APIs autenticadas e autorizadas por perfil, e conectar o front-end a esse dominio para encerrar o uso de persistencia local nas jornadas principais.

  • Implantar infraestrutura de banco, storage e backups como base para todas as sprints seguintes.
  • Modelar entidades core (usuario, paciente, pedido, item, ingrediente, status) e expor endpoints REST.
  • Entregar autenticacao server-side e matriz de autorizacao multi-perfil.
  • Sincronizar kanban da cozinha e jornada do paciente com dados reais do servidor.
  • Cobrir com QA os fluxos criticos de autenticacao, CRUD de pacientes e pedido ponta a ponta.

5. Estrutura de acompanhamento

5.1 To Do

  • Itens que ja possuem descricao, owner sugerido e criterio de aceite definido no GitHub Project.
  • Bloqueadores conhecidos (ex: provisionamento do banco) devem ser explicitos na issue.

5.2 Doing

  • Item em execucao deve apontar escopo atual e pendencias para o criterio de aceite.
  • Evidencias parciais (migrations aplicadas, print de endpoint) podem ser anexadas ao longo do desenvolvimento.

5.3 Done

  • So entra em concluido o item com criterio de aceite cumprido e evidencia registrada.
  • Migrations aplicadas em ambiente de homologacao fazem parte da evidencia.

6. Definition of Done da sprint

  • Modulos back-end publicados, com migrations aplicadas e documentadas.
  • Endpoints autenticados e autorizados conforme matriz de perfis.
  • Jornada nutri-paciente funcional ponta a ponta com API real.
  • Kanban da cozinha operando sobre dados persistidos.
  • Pipeline de CI/CD com deploy validado para front e back.
  • Evidencia minima registrada por issue no GitHub Project.

9. Ritmo de acompanhamento

  • Daily curta para sincronizar front e back em integracoes criticas.
  • Revisao de bloqueios no meio da sprint para priorizar destravamento.
  • Consolidacao incremental de evidencias (prints, logs de migration, gravacao de endpoint).
  • Atualizacao diaria do GitHub Project como fonte unica de verdade.

10. Resultado esperado ao fim da Sprint 3

Ao final da Sprint 3 o FastInBox deve ter migrado a jornada principal para API real: autenticacao, pacientes, pedidos e status sendo lidos e escritos em banco PostgreSQL, com autorizacao por perfil e ambiente publicado. Isso habilita a Sprint 4 a integrar pagamento real sem reabrir dominios anteriores.

Este documento deve ser lido em conjunto com o backlog publicado no GitHub Project e com o documento de evidencias correspondente, gerado ao final da sprint.
