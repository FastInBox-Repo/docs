1. Como ler este documento

Cada bloco abaixo representa uma entrega da Sprint 2. Cada entrega mostra:

  • O que foi feito: funcionalidade tecnica construida e valor gerado.
  • Como foi testado: caminho que a equipe percorreu para validar em baseline.
  • Como voce pode testar: passo a passo direto no ambiente publico.
  • Prints de evidencia: capturas reais feitas automaticamente via Chrome headless a partir do front em localhost:3001, com dados de seed controlados.
As capturas foram geradas pelo script front/scripts/capture-evidencias.mjs, que popula localStorage com dados de demonstracao e navega em sequencia pelos perfis nutricionista, cozinha, admin e paciente.

4. Evidencia visual por entrega

Cada card abaixo combina descricao tecnica, roteiro de validacao e print real do baseline executado nesta sprint.

4.1 Landing publica em modo producao

FRONT | PUBLIC
Rota: / Perfil: visitante Canal: desktop

Home limpa, sem referencia tecnica a protocolos ou ambiente de teste, destacando proposta de valor, chamada para acessar a plataforma e links para documentacao. A tipografia e a composicao seguem o mesmo idioma visual neo-brutal deste hub.

Como foi testado

  1. Build next build finalizou sem erros de lint ou tipagem.
  2. Dev server em localhost:3001 renderizou a pagina publica.
  3. Captura automatica via CDP validou dom e tipografia.

Como voce testa

  1. Abra front-chi-flame.vercel.app.
  2. Confirme que a landing carrega sem mencoes de ambiente de teste.
  3. Acione a chamada principal para chegar ao fluxo de login.

4.2 Tela de login por perfil

FRONT | AUTH
Rota: /login?role=nutricionista Perfil: todos Instrumentado no audit

Login unificado com troca de perfil. A partir desta sprint, cada autenticacao bem sucedida gera evento do tipo login e cada falha registra login_failed no auditStore, viabilizando rastreabilidade minima.

Como foi testado

  1. Login com credenciais validas em cada perfil.
  2. Login com credencial invalida para validar evento de falha.
  3. Verificacao dos eventos na pagina /admin/auditoria.

Como voce testa

  1. Na home publica, acione Acessar plataforma.
  2. Entre como nutricionista (ana@nutritionvida.com.br / 123456).
  3. Depois alterne para admin@fastinbox.com.br e abra /admin/auditoria.

4.3 Area do nutricionista

FRONT | NUTRICIONISTA
Rotas: /nutricionista/dashboard, /nutricionista/pacientes Perfil: Nutricionista

Dashboard da nutricionista com visao de indicadores da clinica e lista de pedidos recentes, mais listagem de pacientes com totais por paciente. As informacoes sao reativas ao sprintStore, entao qualquer pedido criado em outra aba aparece aqui sem precisar recarregar.

Como foi testado

  1. Criar pedido em uma aba usando perfil nutricionista.
  2. Abrir nova aba na mesma origem e validar que os indicadores e lista estavam atualizados.
  3. Conferir evento order_created em /admin/auditoria.

Como voce testa

  1. Logue como nutricionista.
  2. Abra /nutricionista/dashboard e /nutricionista/pacientes.
  3. Cadastre um paciente ou um pedido e veja os contadores reagirem.

4.4 Kanban da cozinha com sincronizacao ao vivo

FRONT | COZINHA
Rota: /cozinha/dashboard Perfil: Fabrica Novo: indicador de sync

O painel da cozinha recebeu indicador de sincronizacao (icone pulsante com timestamp da ultima atualizacao) e responde a eventos do store. Se o paciente confirmar pagamento em outra aba, o pedido aparece na coluna correspondente em tempo real, atendendo ao item Kanban sincronizado do planning.

Como foi testado

  1. Abrir o kanban como cozinha e, em paralelo, pagar um pedido em outra aba.
  2. Validar que o card migra de coluna sem refresh manual.
  3. Verificar que o timestamp de sync atualiza.

Como voce testa

  1. Logue como cozinha (fabrica@fastinbox.com.br / 123456).
  2. Abra outra aba, logue como paciente e confirme um pagamento pendente.
  3. Volte ao kanban e observe o card reposicionar.

4.5 Dashboard administrativo com eventos recentes

FRONT | ADMIN
Rota: /admin/dashboard Perfil: Admin

Dashboard admin agora consome dados reais do sprintStore e exibe uma coluna adicional Eventos recentes com os ultimos seis eventos do auditStore, ligada diretamente ao console de auditoria. As metricas passaram a refletir pedidos realmente criados no ambiente.

Como foi testado

  1. Popular dados atraves do fluxo nutricionista + paciente + cozinha.
  2. Abrir o dashboard admin e validar indicadores.
  3. Clicar em Ver todos e chegar em /admin/auditoria.

Como voce testa

  1. Logue como admin (admin@fastinbox.com.br / 123456).
  2. Abra /admin/dashboard.
  3. Confirme que os eventos na lateral refletem acoes recentes.

4.6 Console de auditoria

FRONT | OBSERVABILIDADE
Rota: /admin/auditoria Perfil: Admin Cap: 200 eventos

Pagina dedicada a leitura do auditStore. Suporta filtros por tipo de evento (login, criacao de pedido, pagamento, mudanca de status, etc.), por ator, busca textual, export em JSON e acao Limpar log. Cada evento mostra icone, descricao, ator, target, metadata e tempo relativo.

Como foi testado

  1. Disparar eventos manuais (login, falha, novo pedido, pagamento, mudanca).
  2. Validar aparecimento em ordem decrescente.
  3. Baixar o JSON exportado e conferir os campos criticos.

Como voce testa

  1. Logue como admin e abra /admin/auditoria.
  2. Aplique filtros, faca busca por codigo do pedido.
  3. Clique em Exportar JSON e valide o download.

4.7 Pagina de diagnostico com smoke test E2E

QA | SMOKE
Rota: /admin/diagnostico 7 passos sequenciais Resultado verde

Nova pagina que executa em ciclo curto o caminho critico do MVP: hidratacao, autenticacao, cadastro de paciente, criacao de pedido, pagamento, avanco no kanban e cleanup. Cada passo tem feedback individual (pending, running, ok, fail) e um sumario com passos verdes e duracao total.

Como foi testado

  1. Executado no baseline da sprint com todos os passos em verde.
  2. Teste forcando falha em um passo para validar estado fail.
  3. Reexecucao com cleanup ativo para garantir que o log nao contamina.

Como voce testa

  1. Logue como admin.
  2. Abra /admin/diagnostico.
  3. Clique em Rodar smoke test e aguarde o sumario final.

4.8 Jornada do paciente com feedbacks reforcados

FRONT | PACIENTE | MOBILE
Rotas: /paciente, /paciente/pedido/<codigo>/status, /paciente/pedido/<codigo>/pagamento Canal: mobile

O fluxo do paciente foi ajustado para responder a um dos itens centrais do planning: feedbacks de interface. A tela de status exibe timeline com horario real da transicao (puxado do audit log), a landing do paciente lista pedidos ativos com selo de urgencia e o pagamento agora passa por estagios explicitos (verificando, processando, confirmando, sucesso) antes de liberar o pedido para a fabrica.

Como foi testado

  1. Emulacao mobile 420x820 no headless Chrome.
  2. Fluxo completo de pagamento com os 4 estagios.
  3. Consulta de status com etapas espelhando audit log.

Como voce testa

  1. Logue como paciente (carla.menezes@email.com / 123456).
  2. Abra um pedido pendente e confirme o pagamento.
  3. Navegue para Acompanhar e veja a timeline com horarios.

6. Como reproduzir localmente

  1. Clonar o repositorio front e rodar npm install.
  2. Subir o dev server com npm run dev na porta 3001.
  3. Opcional: rodar node scripts/capture-evidencias.mjs para regenerar os prints desta pagina.
  4. Visitar o app em http://localhost:3001 e seguir os testes descritos em cada bloco acima.
  5. Para inspecao de observabilidade, logar como admin e abrir /admin/auditoria e /admin/diagnostico.
Todas as interacoes da Sprint 2 funcionam sobre estado persistido em localStorage, com sincronizacao entre abas. A proxima sprint pode promover esse contrato para um backend real sem mudar as superficies ja entregues.
